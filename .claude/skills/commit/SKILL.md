# commit: 변경사항 분석 후 컨벤션 커밋

현재 작업 디렉토리의 git 변경사항을 분석하여 논리적 단위로 나누고, 컨벤션에 맞는 한국어 커밋 메시지를
작성한 뒤 **사용자 승인을 받아** 커밋한다. 사용자 승인 없이 임의로 커밋하지 않는다.

## 절차

### Step 1: 변경사항 분석

다음을 병렬로 실행해 현재 상태를 파악한다.

- `git status` — 추적되지 않은 파일 포함 전체 상태 확인 (`-uall` 플래그는 대형 저장소에서 메모리 문제를
  일으킬 수 있으므로 사용하지 않는다)
- `git diff` — 스테이징되지 않은 변경사항
- `git diff --staged` — 이미 스테이징된 변경사항
- `git log --oneline -10` — 최근 커밋 메시지 스타일(컨벤션·언어·포맷) 파악

변경사항이 전혀 없으면(스테이징/비스테이징/추적되지 않은 파일 모두 없음) 그 사실을 알리고 종료한다.
git 저장소가 아니면 그 사실을 알리고 종료한다.

### Step 2: 논리적 단위로 분류

전체 diff를 하나의 커밋으로 뭉치지 말고, 서로 다른 목적의 변경은 분리한다. 예를 들어:

- 기능 추가와 무관한 리팩터링이 섞여 있으면 분리 고려
- 설정 파일 변경과 코드 변경이 무관하면 분리 고려
- 단, 하나의 논리적 작업이 여러 파일에 걸쳐 있다면(예: 컴포넌트 추가 + 해당 테스트) 하나로 묶는다

분리가 애매하거나 사용자가 이미 "이건 한 커밋으로" 같은 맥락을 준 경우 과도하게 쪼개지 않는다. 커밋 단위
분류 결과를 사용자에게 보여주고 판단할 수 있게 한다.

### Step 3: 커밋 메시지 작성

각 커밋 단위마다 다음 형식으로 한국어 커밋 메시지를 작성한다.

```
<type>: <한 줄 요약>

<선택: 본문 — 왜 이 변경이 필요한지, 무엇을 바꿨는지 1-3문장>
```

**type 종류**:
- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `refactor`: 동작 변경 없는 코드 구조 개선
- `chore`: 빌드/설정/의존성 등 잡무성 변경
- `docs`: 문서만 변경
- `test`: 테스트 추가/수정
- `style`: 포맷팅 등 코드 의미에 영향 없는 변경

**작성 원칙**:
- 요약은 무엇을 바꿨는지보다 **왜** 바꿨는지에 초점 (diff만 봐도 알 수 있는 내용을 반복하지 않는다)
- 저장소의 기존 커밋 메시지 스타일(Step 1의 `git log` 결과)이 있다면 그 컨벤션을 우선 따른다
- 요약은 간결하게 (한 문장, 50자 내외 권장)
- 시크릿이나 민감한 내용이 포함된 파일(`.env`, `credentials.json` 등)은 커밋 대상에서 제외하고
  사용자에게 경고한다

### Step 4: 사용자 승인

작성한 커밋 메시지(들)와 각 커밋에 포함될 파일 목록을 사용자에게 제시하고 승인을 받는다. 선택지:

- **전체 승인**: 제시한 대로 모두 커밋
- **일부 수정**: 메시지나 파일 분류 조정 후 재확인
- **취소**: 커밋하지 않고 종료

사용자가 명시적으로 승인하기 전에는 실제 `git commit`을 실행하지 않는다.

### Step 5: 커밋 실행

승인된 내용대로 파일을 스테이징하고 커밋한다.

- 파일은 이름을 명시해서 추가한다(`git add -A`, `git add .` 금지 — 의도치 않은 파일 포함 방지)
- 커밋 메시지는 헤레독으로 전달해 포맷을 보존한다
- 커밋 메시지 끝에 다음을 추가한다:

```
Co-Authored-By: Claude <noreply@anthropic.com>
```

- pre-commit 훅이 실패하면 원인을 파악해 수정한 뒤 **새 커밋**을 생성한다(기존 커밋을 `--amend`하지 않는다.
  훅 실패 시점엔 커밋이 아직 생성되지 않았으므로 `--amend`는 직전 커밋을 덮어써 버린다)
- 커밋 후 `git status`로 성공 여부를 확인한다

## 주의사항

- `git push`는 사용자가 명시적으로 요청하지 않는 한 실행하지 않는다
- `--no-verify`, `--no-gpg-sign` 등 훅·서명을 건너뛰는 옵션은 사용자가 명시적으로 요청하지 않는 한
  사용하지 않는다
- `git reset --hard`, `git checkout --`, `git clean -f` 등 파괴적 명령은 이 스킬의 범위에 없다
- 커밋 대상 파일에 시크릿으로 의심되는 내용이 있으면 파일명이 무관해 보여도 내용을 확인하고 사용자에게
  경고한다
